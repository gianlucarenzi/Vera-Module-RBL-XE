% --- SEZIONE 1: INTRODUZIONE ---
\section{Introduzione}
Questo documento presenta un'analisi tecnica del firmware per il progetto \textit{Vera-Module-RBL-XE}. L'obiettivo del firmware, eseguito su un microcontrollore ESP32, è di interfacciarsi con il bus di un processore 6502 per emulare un dispositivo PBI (Parallel Bus Interface) o CCTL (Cartridge Control Line) per computer ATARI della serie XL/XE.

Il progetto supporta due MCU target e due modalità operative, per un totale di quattro varianti di compilazione distinte. L'analisi si concentra sulla fattibilità temporale (timing) dell'interazione tra la MCU moderna e la CPU vintage, illustrando la logica di funzionamento tramite diagrammi di flusso e temporali.

% --- SEZIONE 2: ARCHITETTURA DEL SISTEMA ---
\section{Architettura del Sistema}
Il sistema si basa sull'interazione di due componenti principali.

\subsection{Componenti Chiave}
\begin{itemize}
    \item {\bfseries CPU 6502 (MOS/SALLY):} Il processore centrale dell'Atari, operante a una frequenza di clock di \textbf{1.7897 MHz} (NTSC).
    \item {\bfseries MCU (selezionabile a compile-time):} Il microcontrollore moderno responsabile del monitoraggio del bus e dell'emulazione della periferica. Sono supportati due target:
    \begin{itemize}
        \item \textbf{ESP32-S3FN8} (QFN56, 7$\times$7\,mm) — Dual-core Xtensa LX7 @ 240\,MHz, 512\,KB SRAM, 8\,MB flash in-package Quad SPI, 45 GPIO bidirezionali.
        \item \textbf{ESP32-WROOM-32 / NodeMCU DevKit V1} (ESP32-PICO-D4) — Dual-core Xtensa LX6 @ 240\,MHz, 520\,KB SRAM, 4\,MB flash, GPIO34--39 solo input.
    \end{itemize}
\end{itemize}

\subsection{Modalità operative (BUS\_MODE)}
Il firmware supporta due modalità operative, selezionate a compile-time tramite la macro \texttt{BUS\_MODE}:

\begin{itemize}
    \item \textbf{PBI mode (\texttt{BUS\_MODE=0}):} Il modulo si collega al connettore ECI della serie XL/XE. Il firmware decodifica la pagina \$D1xx tramite il segnale \texttt{D1XX\_N}, serve una ROM da 2\,KB nell'area \$D800--\$DFFF (segnale \texttt{ROM\_SEL\_N} dalla 74HC138), e controlla \texttt{EXTSEL\_N} per disabilitare la ROM floating-point dell'Atari quando il dispositivo viene selezionato.
    \item \textbf{CCTL mode (\texttt{BUS\_MODE=1}):} Il modulo si collega alla porta cartuccia. Il firmware decodifica i 5 bit bassi dell'indirizzo nell'area \$D500--\$D5FF tramite il segnale \texttt{CCTL\_N} e genera il chip-select per il chip VERA.
\end{itemize}

\subsection{Ambienti di build PlatformIO}
Il file \texttt{platformio.ini} definisce quattro ambienti indipendenti:

\begin{center}

La selezione dell'MCU avviene tramite le macro \texttt{MCU\_ESP32\_S3} e \texttt{MCU\_ESP32\_CLASSIC} passate come \texttt{-D} nei \texttt{build\_flags}.

\subsection{Interfacciamento Bus}
L'ESP32 è collegato direttamente alle seguenti linee del bus del 6502 tramite tre level-shifter bidirezionali TXS0108E (3.3\,V lato MCU, 5\,V lato Atari):
\begin{itemize}
    \item {\bfseries Bus Dati (D0--D7):} Bidirezionale; la direzione è controllata dal segnale \texttt{R/W\_}.
    \item {\bfseries Bus Indirizzi (A0--A10):} Solo input; utilizzati per decodificare l'area di memoria di interesse. In modalità CCTL vengono usati solo A0--A4.
    \item {\bfseries Segnali di controllo:} \texttt{PHI2} (clock CPU), \texttt{R/W\_} (lettura/scrittura), \texttt{D1XX\_N} o \texttt{CCTL\_N} (selezione pagina), \texttt{ROM\_SEL\_N} (solo PBI), \texttt{EXTSEL\_N} e \texttt{DEV\_SEL\_N} (output verso l'Atari e verso VERA).
\end{itemize}

% --- SEZIONE 4: ANALISI DELLE TEMPISTICHE ---
\section{Analisi delle Tempistiche}
Una delle principali preoccupazioni in un progetto di questo tipo è la capacità della MCU di rispondere entro la finestra temporale definita dal ciclo di clock della CPU 6502.

\begin{itemize}
    \item {\bfseries Ciclo Clock 6502 (1.7897 MHz, NTSC):} $T_{6502} = 1 / (1.7897 \times 10^6) \approx 559\,\text{ns}$
    \item {\bfseries Finestra PHI2 alto (half-cycle):} $T_{PHI2\uparrow} \approx 279\,\text{ns}$ — questa è la finestra critica entro cui l'ESP32 deve leggere l'indirizzo, decodificarlo ed eventualmente pilotare il bus dati.
    \item {\bfseries Ciclo Clock ESP32 (240 MHz):} $T_{ESP32} = 1 / (240 \times 10^6) \approx 4.17\,\text{ns}$
\end{itemize}

Il rapporto tra la finestra critica e il ciclo dell'ESP32 è:
\[
\frac{T_{PHI2\uparrow}}{T_{ESP32}} = \frac{279\,\text{ns}}{4.17\,\text{ns}} \approx 67\,\text{cicli ESP32}
\]

Anche considerando solo la semi-finestra di PHI2 alto (il caso peggiore), l'ESP32-S3 dispone di circa \textbf{67 cicli di clock} per completare tutta la catena: lettura di \texttt{GPIO.in} / \texttt{GPIO.in1.val}, decodifica dell'indirizzo, lookup nella LUT, pilotaggio del bus dati. Poiché tutte le funzioni critiche risiedono in IRAM (\texttt{IRAM\_ATTR}) ed operano solo su registri hardware, il percorso critico è tipicamente inferiore a 20 cicli.

\subsection{Accesso diretto ai registri GPIO}
Per eliminare qualsiasi overhead di astrazione, il firmware usa esclusivamente accesso diretto ai registri hardware dell'ESP32:

\begin{itemize}
    \item \texttt{GPIO.in} — legge i 32 GPIO del bank-0 in un'unica istruzione a 32 bit.
    \item \texttt{GPIO.in1.val} — legge i GPIO del bank-1 (GPIO32--48).
    \item \texttt{GPIO.out\_w1ts} / \texttt{GPIO.out\_w1tc} — set/clear atomico dei bit di uscita (write-1-to-set / write-1-to-clear).
    \item \texttt{GPIO.enable\_w1ts} / \texttt{GPIO.enable\_w1tc} — abilitazione/disabilitazione in uscita (tristate) per il bus dati.
\end{itemize}

% --- SEZIONE 5: CODICE SORGENTE ---
\section{Frammento di Codice Sorgente}
Di seguito è riportato un estratto del file \texttt{main.cpp} che mostra la struttura di selezione a compile-time della MCU e il cuore del \texttt{MonitorTask} in modalità PBI. Tutte le funzioni critiche sono marcate \texttt{IRAM\_ATTR} per garantire l'esecuzione dalla RAM interna, indipendentemente dallo stato della cache flash.

\begin{lstlisting}[style=cppstyle, caption={Selezione MCU e decode per ESP32-S3FN8 in main.cpp}, label=code:mcu_select]
#if defined(MCU_ESP32_S3)
  #define PIN_PHI2 2   #define PIN_RW 17
  #define PIN_BUS_SEL_N 18  #define PIN_ROM_SEL_N 21
  #define PIN_EXTSEL_N  0   #define PIN_DEV_SEL_N  1
  #define PIN_UART_TX  43
  #define DBUS_MASK \
    ((1UL<<4)|(1UL<<5)|...|(1UL<<11))   /* GPIO4-11, 0x00000FF0 */
  static const uint8_t DBUS_PINS[8] = {4,5,6,7,8,9,10,11};

  /* A0-A5: GPIO.in1.val bits 1-6; A6-A10: GPIO.in bits 12-16 */
  static inline uint16_t IRAM_ATTR decode_addr(uint32_t lo, uint32_t hi) {
    uint16_t a = (uint16_t)((hi >> 1) & 0x3Fu);     /* A0-A5 */
    a |= (uint16_t)(((lo >> 12) & 1u) <<  6);        /* A6    */
    /* ... A7-A10 analoghi ... */
    return a;
  }
  static inline uint8_t IRAM_ATTR decode_data(uint32_t lo) {
    return (uint8_t)((lo >> 4) & 0xFFu); /* GPIO4-11 compatti */
  }

#elif defined(MCU_ESP32_CLASSIC)
  /* Pin e decode differenti per NodeMCU / PICO-D4 */
  #define PIN_RW  15  #define PIN_BUS_SEL_N 5
  /* D0-D7: GPIO18,19,21-23,25-27 (non contigui) */
  static inline uint8_t IRAM_ATTR decode_data(uint32_t lo) {
    return (uint8_t)(
      ((lo >> 18) & 0x03u) |
      (((lo >> 21) & 0x07u) << 2) |
      (((lo >> 25) & 0x07u) << 5));
  }
#else
  #error "Set -D MCU_ESP32_S3 or -D MCU_ESP32_CLASSIC"
#endif
\end{lstlisting}

\begin{lstlisting}[style=cppstyle, caption={Nucleo del MonitorTask (modalità PBI, semplificato)}, label=code:monitor_task]
static void IRAM_ATTR MonitorTask(void *arg)
{
    bool selected = false;
    bus_release();    /* tristate data bus */

    for (;;)
    {
        /* 1. Attesa fronte di salita di PHI2 + snapshot */
        uint32_t g_lo;
        while (!((g_lo = GPIO.in) & (1UL << PIN_PHI2)));

        /* 2. Decodifica segnali e indirizzo */
        bool is_read  = (g_lo & (1UL << PIN_RW))        != 0;
        bool is_d1xx  = (g_lo & (1UL << PIN_BUS_SEL_N)) == 0;
        bool is_rom   = (g_lo & (1UL << PIN_ROM_SEL_N))  == 0;
        uint16_t addr = decode_addr(g_lo, GPIO.in1.val);
        uint8_t  off  = (uint8_t)(addr & 0xFFu);

        /* 3. Selezione dispositivo (scrittura D1FF) */
        if (!is_read && is_d1xx && off == 0xFFu) {
            uint8_t data = decode_data(GPIO.in);
            bool new_sel = (data == DEVICE_MASK);   /* 0x80 */
            if (new_sel != selected) {
                selected = new_sel;
                /* EXTSEL_N alto = ROM FP attiva; basso = disattivata */
                if (selected) GPIO.out_w1tc = (1UL << PIN_EXTSEL_N);
                else          GPIO.out_w1ts = (1UL << PIN_EXTSEL_N);
            }
        }
        /* 4. Lettura ROM ($D800-$DFFF) */
        if (is_read && selected && is_rom) {
            bus_drive(pbi_driver[addr & 0x7FFu]);
        }

        /* 5. Attesa fine PHI2, rilascio bus */
        while (GPIO.in & (1UL << PIN_PHI2));
        bus_release();
        GPIO.out_w1ts = (1UL << PIN_DEV_SEL_N); /* deassert VERA CS */
    }
}
\end{lstlisting}

% --- SEZIONE 7: CONCLUSIONE ---
\section{Conclusione}
L'analisi conferma che l'architettura hardware e software del progetto è solida e performante. L'ESP32-S3FN8 a 240\,MHz garantisce circa 67 cicli di clock nella finestra critica di PHI2 alto (279\,ns), ampiamente sufficienti per l'intera catena di decodifica e pilotaggio del bus. Le tecniche di programmazione a basso livello — accesso diretto ai registri GPIO (\texttt{GPIO.in}, \texttt{GPIO.in1.val}, \texttt{GPIO.out\_w1ts}/\texttt{w1tc}), funzioni in IRAM, LUT pre-calcolata per il bus dati — garantiscono un percorso critico tipicamente inferiore a 20 cicli.

Il supporto dual-MCU (\texttt{MCU\_ESP32\_S3} / \texttt{MCU\_ESP32\_CLASSIC}) e dual-mode (\texttt{BUS\_MODE=0} PBI, \texttt{BUS\_MODE=1} CCTL) rende il firmware portabile su hardware diverso senza duplicazione di codice. La tabella dei pin fisici del package QFN56 consente di realizzare il PCB direttamente dalla documentazione senza ambiguità tra numero GPIO logico e pad fisico.
